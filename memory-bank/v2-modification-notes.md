## 1. 文档信息

| 项目 | 内容 |

| 文档版本 | v2.0 修改意见 |

## 2. 背景与问题

当前系统已经跑通 Collection、Analysis、QA、Writer 多 Agent 链路，并具备竞争图谱、产品画像、报告、过程追踪和 Markdown 导出能力。但手动测试后发现，现有产品表达更像“结构化数据和 Agent 产物陈列”，而不是“给产品经理做决策的竞品分析工作台”。

主要问题如下：

1. 页面以卡片堆砌为主，缺少清晰的信息层级和阅读脉络。
2. 多数内容直接展示 Product、Claim、Evidence、Trace 等结构化字段，适合系统调试，不适合产品经理阅读。
3. 分析解释不足，缺少“这意味着什么”“为什么这么判断”“下一步该做什么”。
4. 页面和报告中仍存在大量英文标题或裸技术词，例如 `FeatureTree`、`PricingModel`、`Trace`、`Agent Run`、`Markdown`。
5. 报告导出仍为 Markdown，不满足正式汇报和交付质感。
6. 系统能力展示和产品决策阅读混在一起，导致主线被 Agent、Token、Tool Call、Payload 等细节打断。

2.0 改版需要从“展示 AI 产物”转向“支持产品经理做竞争决策”，同时保留可下钻的证据链、QA 打回和智能体过程，以证明系统能力。

## 3. 产品定位

2.0 的产品定位是：面向产品经理的竞品决策工作台。

默认展示层应该帮助产品经理快速判断：

1. 当前目标产品处在什么竞争战场。
2. 谁是最需要盯防的竞品。
3. 目标产品赢在哪里、输在哪里。
4. 当前结论能不能用于初步决策。
5. 下一步产品、表达、价格或证据补齐动作是什么。

系统能力不应被隐藏，但应作为可下钻层存在。用户需要时，可以从任意关键结论进入对应的分析逻辑、证据来源、QA 审查、智能体过程和差异记录。

## 4. 核心目标

1. 让产品经理先看到可决策结论。
2. 让每个结论都能下钻到分析逻辑和证据。
3. 让正式报告可以通过网页打印和 Word `.docx` 交付。

## 5. 用户角色

| 用户 | 优先级 | 诉求 |
|---|---:|---|
| 产品经理 | P0 | 快速理解竞争格局、机会、风险和下一步动作 |
| 参赛评委 | P1 | 看到系统具备多 Agent、证据链、QA 打回和可追溯能力 |
| 团队成员 | P1 | 复核分析依据、定位证据问题、追踪工作流过程 |
| 技术查看者 | P2 | 查看 Agent Run、Tool Call、Token、Diff 等底层细节 |

## 6. 设计原则

### 6.1 先判断，再解释，再下钻

页面第一层展示判断和建议，不默认展示原始字段列表。每个判断旁边提供“查看依据”“查看过程”等入口。

### 6.2 给人看，不给 AI 看

主界面必须使用自然中文表达。技术字段、枚举、ID、Agent 名称、Evidence ID 等只能作为下钻信息出现。

### 6.3 结论必须带可信状态

所有关键结论必须标明判断强度、证据风险和决策可用状态，避免把推断当事实。

### 6.4 系统能力可见但不抢主线

QA 打回、证据补齐、智能体过程、Diff 等能力保留，但默认折叠或放在二级 Tab。

### 6.5 禁止卡片堆砌

2.0 页面不得只由卡片列表构成。每个核心页面必须包含：

1. 主结论区。
2. 分析解释区。
3. 证据或过程下钻区。
4. 明确的下一步动作或页面目的。

## 7. 信息架构

2.0 导航调整为：

1. 竞争态势总览
2. 竞争图谱
3. 产品与竞品画像
4. 分析报告
5. 证据与过程追踪

创建任务后默认进入“竞争态势总览”。过程追踪不再作为默认落点，但保留为证据和系统能力追溯入口。

## 8. 页面需求

### 8.1 竞争态势总览

#### 页面目标

让产品经理在首屏理解当前竞争判断、决策可用性、关键竞品、首要建议和证据风险。

#### 首屏必须展示

1. 一句话判断。
2. 决策可用状态。
3. 关键竞品缩略图。
4. 首要行动建议。
5. 证据风险提示。

#### 一句话判断

默认使用判断型表达，但必须带证据风险提示。

示例：

> 目标产品当前最大问题不是功能覆盖，而是核心竞品在安全可信和价格解释上更容易促成下单。

判断强度分为：

| 等级 | 含义 |
|---|---|
| 明确判断 | 证据较完整，结论可用于初步决策 |
| 倾向判断 | 证据支持方向，但仍有局限 |
| 仅作假设 | 证据不足，只能作为后续验证方向 |

#### 决策可用状态

总览页必须给出本次分析能否用于决策的总状态，并显示原因。

| 状态 | 含义 |
|---|---|
| 可用于初步决策 | 关键结论有证据支持，QA 风险已解决或可接受 |
| 建议谨慎决策 | 存在证据缺失、低可信或未完全解决的 QA 风险 |
| 仅供方向参考 | 核心结论证据不足，不建议直接用于产品决策 |

#### 关键竞品

默认展示 3 个对象：

1. 最高威胁直接竞品。
2. 最高威胁替代竞品。
3. 需复核高分竞品。

如果某类对象不存在，不强行补齐。每个竞品展示：

1. 产品缩略图。
2. 产品名称。
3. 关系标签。
4. 威胁等级。
5. 证据可信状态。
6. “查看竞争关系”入口。

#### 分析范围

总览页必须展示本次分析范围：

1. SKU 数量。
2. 数据来源。
3. 品类和子类。
4. 快照日期或访问时间范围。
5. 是否为实时数据。

若字段缺失，显示“暂无可靠数据”。必须明确提示：本报告基于用户提供的脱敏 SKU 快照，不代表实时全网数据。

#### 切片控件

总览页下方保留轻量切片控件：

1. 价格带。
2. 用户人群。
3. 使用场景。

切片切换后，一句话判断、关键竞品、机会点、风险点和行动建议都必须同步刷新。

#### 机会点与风险点

机会点最多 3 条，分为：

1. 产品机会：功能、体验、定位相关。
2. 表达机会：主图、详情页、卖点、证据呈现相关。

风险点最多 3 条，分为：

1. 证据风险。
2. 竞争风险。
3. 表达风险。
4. 合规风险。

#### 行动建议

行动建议最多 5 条，必须分优先级：

1. P0 立即处理。
2. P1 本轮优化。
3. P2 后续验证。

每条建议必须标明责任类型：

1. 产品功能。
2. 商品详情页/内容表达。
3. 价格策略。
4. 证据补齐/调研验证。

### 8.2 竞争图谱

#### 页面目标

展示目标产品所在竞争战场，并解释为什么某些对象构成关键威胁。

#### 默认展示

默认只展示 3 到 5 个关键竞争关系。关键关系由后端筛选，并返回入选理由。筛选逻辑采用混合策略：

1. 最高威胁直接竞品。
2. 最高威胁替代竞品。
3. 需复核高分竞品。
4. 对产品策略动作最有启发的竞品。

页面提供“展开全部关系”的开关。

#### 关系标签

在原有竞争类型基础上，增加 PM 更容易理解的关系标签：

| 标签 | 说明 |
|---|---|
| 正面硬碰 | 同类产品在同一价格、人群、场景中直接竞争 |
| 低价截流 | 竞品通过更低到手价影响用户决策完成 |
| 场景替代 | 竞品不完全同类，但解决同一使用场景问题 |
| 信任压制 | 竞品在安全、认证、评价、售后等信任因素上更强 |
| 内容种草竞争 | 竞品通过卖点表达或内容共现影响信息触达和兴趣形成 |

每个标签必须提供简短解释。

#### 威胁等级

默认展示威胁等级，弱化裸分数。

| 等级 | 规则 |
|---|---|
| 高威胁 | 竞争分大于等于 0.80 且证据可信 |
| 中威胁 | 竞争分 0.60 到 0.79 且证据基本可用 |
| 低威胁 | 竞争分低于 0.60 |
| 高分需复核 | 分数较高但存在关键证据缺失、低可信或未解决 QA 风险 |

如果证据不足，即使竞争分高，也不能直接标为“高威胁”，必须展示“高分需复核”。

#### 竞争边详情

每条竞争关系采用四段式解释：

1. 为什么它是竞品。
2. 它强在哪里。
3. 它会在哪个决策阶段抢走用户。
4. 我们该怎么应对。

“我们该怎么应对”允许生成具体商品表达建议，但必须标记为分析建议，不能新增无证据事实。

示例：

> 建议在详情页优先解释防夹检测和异常停止机制，而不是只写“安全无忧”。

#### 五维评分展示

五维评分名称保留原有说法，但每个维度增加一句解释：

| 维度 | 解释 |
|---|---|
| 需求替代性 | 判断用户会不会把它当成同一个购买需求的替代选择 |
| 上下文匹配度 | 判断它是否匹配当前价格带、人群和使用场景 |
| 决策阶段影响力 | 判断它会在哪个购买阶段影响用户下单 |
| 证据置信度 | 判断支撑该关系的证据够不够完整可靠 |
| 市场信号强度 | 判断评论、评分、内容共现等市场反馈是否足够强 |

### 8.3 产品与竞品画像

#### 页面目标

从单一目标产品画像升级为横向对比，帮助产品经理理解目标产品相对核心竞品的优势、短板和证据状态。

#### 默认对比对象

默认展示 3 列：

1. 目标产品。
2. 最高威胁直接竞品。
3. 最高威胁替代竞品。

用户可以展开更多竞品。

#### 第一屏维度

第一屏必须展示：

1. 价格带。
2. 核心卖点。
3. 主要人群。
4. 使用场景。
5. 证据可信状态。

功能树、评论痛点、截图证据放在下钻层。

#### 优劣判断

横向对比每个关键维度需要高亮目标产品状态：

1. 优势。
2. 持平。
3. 短板。
4. 证据不足。

这些判断必须可下钻查看依据。

#### 人工修正

保留受控人工修正能力，但命名改为面向业务的表达：

1. 修正画像。
2. 标记不采纳。
3. 补充证据备注。

不允许自由编辑整份报告或直接改写 Claim 正文。所有修正仍走 Human Feedback 受控接口，并保留 before/after 和原因。

### 8.4 分析报告

#### 页面目标

提供正式、可阅读、可汇报、可打印、可导出 Word 的竞品分析报告。

#### 视图模式

报告页提供两个模式：

1. 工作台视图：保留下钻、证据、过程入口。
2. 打印视图：隐藏导航、按钮和交互控件，只保留正式报告内容。

报告页顶部提供：

1. 下载 Word 报告。
2. 打印/另存 PDF。
3. 切换工作台视图/打印视图。

总览页右上角也提供下载 Word 报告的次级入口。

#### 报告章节

2.0 报告结构调整为：

1. 结论摘要。
2. 竞争格局判断。
3. 核心竞品拆解。
4. 用户决策链分析。
5. 目标产品机会与风险。
6. 产品策略建议。
7. 证据与质检附录。
8. 分析流程与系统能力附录。

报告不再以 Evidence 索引和 QA 摘要作为主章节标题；这些内容进入附录。

#### 正式报告表达要求

1. 使用自然中文。
2. 先写判断，再写依据。
3. 每个关键判断必须标明判断强度。
4. 每个关键建议必须标明优先级和责任类型。
5. 找不到可靠数据时写“暂无可靠数据”。
6. 推断内容必须明确标记为分析判断或假设。
7. 不得在无证据情况下补价格、认证、尺寸、销量、排名。

#### 静态图谱摘要

打印视图必须包含静态图谱摘要，不能只依赖交互图。静态摘要包括：

1. 3 到 5 条关键竞争关系。
2. 每条关系的威胁等级。
3. 关系标签。
4. 应对建议。
5. 证据可信状态。

### 8.5 证据与过程追踪

#### 页面目标

承接旧版过程追踪能力，但默认按 PM 阅读路径组织证据和质检信息。

#### Tab 结构

1. 证据链。
2. 质检记录。
3. 智能体过程。
4. 差异记录。

默认 Tab 为“证据链”。

#### 证据链

证据链按“结论”组织，而不是按 Evidence 列表组织。

每条结论展示：

1. 结论内容。
2. 判断强度。
3. 证据可信状态。
4. 引用证据数量。
5. 查看依据入口。

展开后展示：

1. Evidence 摘要。
2. 来源类型。
3. 访问时间。
4. 截图缩略图。
5. 证据 ID。
6. 局限性。
7. 风险标记。

#### 质检记录

展示 QA 检查项、问题等级、打回目标、处理结果、是否仍需关注。

#### 智能体过程

页面展示中文名称，并在括号中保留英文原始字段：

1. 采集智能体（collection_agent）
2. 分析智能体（analysis_agent）
3. 质检智能体（qa_agent）
4. 报告智能体（writer_agent）

Token Usage、Tool Call、Payload 等技术详情保留在折叠区，默认不展开。

#### 差异记录

展示 QA 打回或人工修正前后的变化。Diff 必须用业务语言解释变化影响，而不是只展示 before/after JSON。

## 9. 中文化要求

所有用户可见内容必须中文化，包括：

1. 页面主导航。
2. 页面标题。
3. 模块标题。
4. 按钮。
5. 状态。
6. 报告导出内容。
7. 图谱说明。
8. Trace 展示。
9. 错误提示。
10. Mock 数据中的展示文案。

允许保留以下技术缩写：

1. SKU。
2. QA。
3. DOCX。

`Agent`、`Trace` 等词在主界面必须翻译为“智能体”“过程追踪”。展开技术详情时使用中文名称加括号原始字段。

## 10. 术语解释

需要新增统一术语解释组件。以下概念必须提供简短解释：

1. 需求替代性。
2. 上下文匹配度。
3. 决策阶段影响力。
4. 证据置信度。
5. 市场信号强度。
6. QA。
7. 证据可信状态。
8. 动态切片。
9. 威胁等级。
10. 判断强度。

解释文案必须短，不做成说明书。

## 11. 数据与接口需求

### 11.1 新增总览接口

新增：

```text
GET /tasks/{task_id}/overview
```

后端直接返回 PM 可读的总览数据，避免前端从 profile、battlefield、report 拼接。

建议返回内容：

1. 一句话判断。
2. 判断强度。
3. 决策可用状态和原因。
4. 分析范围。
5. 关键竞品。
6. 机会点。
7. 风险点。
8. 行动建议。
9. 当前切片。
10. 下钻引用。

### 11.2 竞争图谱接口升级

`GET /tasks/{task_id}/battlefield` 保留，但返回结构需要补充：

1. 关键关系筛选结果。
2. 入选理由。
3. 威胁等级。
4. PM 关系标签。
5. 四段式解释。
6. 应对建议。
7. 证据可信状态。
8. 产品图片路径。

关键关系排序和筛选由后端完成，前端只负责展示。

### 11.3 图片字段

每个 Product 需要新增或派生：

```text
primary_image_path
```

来源优先级：

1. 快照或素材中绑定的远程商品主图 URL。
2. `data/raw/sku_xx` 中识别出的首图对应的后端可访问 URL。
3. Evidence 的 `screenshot_path` 对应的后端可访问 URL。
4. 远程 URL 不可访问且本地图片也不可用时，显示“暂无可靠图片”。

后端返回给前端的图片字段必须是浏览器可访问 URL，不暴露本地绝对文件路径。

### 11.4 Word 导出接口

新增：

```text
GET /tasks/{task_id}/report/docx
```

删除旧接口：

```text
GET /tasks/{task_id}/report/markdown
```

前端不得再展示 Markdown 导出入口。
Word 导出成功时直接返回真实 `.docx` 文件下载响应；失败时返回标准错误提示，且不得影响网页报告阅读。

`.docx` 文件保存到：

```text
data/reports/<task_id>_<report_id>.docx
```

### 11.5 简化关系图产物

后端生成 Word 报告时，需要生成简化关系图图片：

```text
data/reports/<task_id>_<report_id>_competition_graph.png
```

Word 报告引用该图片。页面打印视图继续展示更漂亮的前端交互图和静态图谱摘要。

### 11.6 推荐新增依赖

允许新增轻量依赖：

1. `python-docx`：生成 `.docx` 报告。
2. `Pillow`：生成简化关系图和处理图片。

不得引入复杂导出服务、Headless Office、Graphviz 系统依赖或后端 PDF 生成服务。

## 12. Word 报告需求

Word 报告必须是真实 `.docx` 文件。

### 12.1 封面

封面包含：

1. 报告标题。
2. 目标产品。
3. 生成时间。
4. 数据来源。
5. 分析范围。
6. 决策可用状态。

数据来源提示必须醒目：

> 本报告基于用户提供的脱敏抖音 SKU 快照，不代表实时全网数据。

### 12.2 目录

Word 报告需要简单静态目录，覆盖主要章节；不要求生成依赖 Office 刷新域的自动目录。

### 12.3 图片

Word 报告需要包含：

1. 目标产品缩略图。
2. 核心竞品缩略图。
3. 后端生成的简化竞争关系图。

如果图片缺失，跳过图片并写“暂无可靠图片”，不得导致整份报告导出失败。

### 12.4 附录

Word 报告末尾包含：

1. 证据与质检附录。
2. 分析流程与系统能力附录。

系统能力附录说明 Collection、Analysis、QA、Writer、QA 打回和证据补齐过程，但不抢主报告篇幅。

## 13. 合规与表达规则

1. 报告不得把推断写成事实。
2. 不得在无证据情况下补价格、认证、尺寸、销量、排名。
3. 对宠物安全、电器认证、医疗美容等敏感内容必须保守。
4. 找不到可靠数据时写“暂无可靠数据”。
5. 单条评论、低可信来源、缺访问时间证据只能作为方向参考。
6. 模型润色不能新增事实，只能改写表达。
7. 模型润色与结构化证据冲突时，以结构化证据和 QA 结果为准。

## 15. 非目标

2.0 不做以下能力：

1. 不做真实外部实时采集。
2. 不做 PPT 导出。
3. 不做后端 PDF 生成。
4. 不做自由编辑整份报告。
5. 不做复杂 BI 图表。
6. 不做多任务协同。
7. 不引入 Celery、Redis、PostgreSQL、Next.js、Redux、Tailwind。
8. 不引入复杂文档服务或外部图渲染服务。

## 16. 验收标准

### 16.1 页面体验

1. 总览页首屏不滚动即可看到一句话判断、决策可用状态、关键竞品缩略图、首要行动建议和证据风险提示。
2. 页面不得只是卡片堆砌，必须有主结论、解释和下钻路径。
3. 所有主导航、标题、按钮、状态、说明文案必须中文化。
4. 技术字段只能在下钻详情中以中文加括号原始字段方式展示。
5. 每个关键结论都可以一键查看依据。
6. 每个关键结论都可以查看相关过程信息。
7. 移动或窄屏下核心文本不重叠、不严重水平溢出。

### 16.2 分析表达

1. 关键结论必须有判断强度。
2. 总览页必须有决策可用状态和原因。
3. 具体结论必须展示可信状态：可直接采纳、谨慎参考或证据不足。
4. 竞争边必须展示四段式解释。
5. 五维评分必须保留原名称并提供一句话解释。
6. 策略建议必须有优先级和责任类型。

### 16.3 证据与过程

1. 证据链默认按结论组织。
2. 证据展开后展示截图缩略图、来源、访问时间、可信度、局限性和证据 ID。
3. QA 打回、证据补齐、Analysis 重算和人工修正必须可追溯。
4. Token Usage、Tool Call、Payload 保留在技术详情折叠区，默认不展开。
5. 导出失败必须写入过程追踪，不影响网页报告阅读。

### 16.4 导出

1. 前端不再展示 Markdown 导出。
2. 后端提供真实 `.docx` 导出。
3. Word 报告包含封面、目录、正文、图片、简化关系图和附录。
4. 图片缺失时显示“暂无可靠图片”，不导致导出失败。
5. 网页报告提供打印视图和浏览器打印按钮。

### 16.5 测试

后续实现需要补充以下测试：

1. 后端 overview API 测试。
2. 后端 docx 导出测试。
3. 后端简化关系图生成测试。
4. 后端图片缺失降级测试。
5. 前端总览页渲染测试。
6. 前端中文化测试。
7. 前端查看依据下钻测试。
8. 前端打印视图测试。
9. Playwright 总览、图谱、报告、证据追踪截图测试。
10. Markdown 接口已删除的回归测试。
