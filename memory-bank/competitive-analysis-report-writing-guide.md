# 竞品分析报告写作规范与自动猫砂盆报告升级指南

## 1. 文档目的

当前系统已经具备 Collection、Analysis、QA、Writer 多 Agent 链路、证据链、QA 打回、Trace、网页报告和 Word 导出能力，但最终报告仍容易显得“不够正式、内容偏少、像 SKU 信息整理”。本指南用于把参考资料中的竞品分析报告写法沉淀成项目内可执行的写作标准，后续 Writer Agent、报告页和 Word 导出都应优先对齐本文档。

本文档不是通用写作课，而是面向本项目“自动猫砂盆竞品分析”场景的报告生成规范。核心目标是让报告从“产品字段罗列”升级为“面向产品经理和运营决策的正式竞品分析报告”。

## 2. 调研范围与样本观察

该目录下包含约 1600 份竞品分析、产品分析、行业分析、方法论、PPT 案例、DOCX 模板和 PDF 案例。由于本轮目标是形成写作规范，抽样重点放在可抽取正文结构的 DOCX/PPTX 样本和目录级样本上，PDF 文件主要用于观察题材分布和报告类型。抽样中重点参考了以下类型：

1. 电商类竞品报告：如京东、淘宝、拼多多、抖音电商、直播电商等。
2. 产品经理方法论文档：如“为什么要做竞品分析”“互联网竞品分析思路及方法”“产品经理如何做好产品竞品分析”等。
3. 行业研究报告：如家居家装、薯片行业、家具行业、互联网金融等。
4. 产品体验/功能对比报告：如移动直播、校园活动、B 端协作工具等。
5. 竞品分析模板：如“XXX 竞品分析报告_V1.0_Build_01_140914.docx”等。

抽样结论：

1. 正式报告不会一上来铺 SKU 字段，而是先交代研究目的、结论、竞争对象选择逻辑和分析范围。
2. 好的竞品报告围绕“为什么比较这些对象、它们在哪些用户决策环节影响目标产品、下一步该怎么做”展开。
3. 行业报告通常先给市场规模、消费结构、市场份额、集中度、政策/技术环境，再进入产品和企业层面的对比。
4. 产品经理类报告强调先明确竞品层级：直接竞品、间接竞品、替代方案、同行业不同模式或潜在威胁。
5. 产品体验类报告常用“战略层、范围层、结构层、框架层、表现层”或“用户路径、功能模块、运营策略、数据表现、优缺点总结”的结构。
6. 正式报告会大量使用矩阵、表格、评分、优先级、行动清单和风险边界，而不是长段堆叠描述。
7. 结论必须服务业务动作：定位、定价、卖点表达、功能补齐、评论痛点修复、证据补采、运营策略调整。

## 3. 一份正式竞品分析报告应回答的问题

自动猫砂盆竞品分析报告至少要回答八个问题：

1. 当前目标产品所处的竞争战场是什么？
2. 哪些产品是真正会被用户拿来比较的竞品？
3. 哪些竞品威胁最大，威胁发生在什么价格带、人群和使用场景？
4. 用户为什么会在这些产品之间犹豫？
5. 目标产品相对竞品强在哪里、弱在哪里、证据是否充分？
6. 当前最值得做的产品、内容、定价或证据动作是什么？
7. 哪些结论是证据支持的，哪些只是推断，哪些暂无可靠数据？
8. QA 打回和人工修正后，报告结论是否仍然可追溯、可复核？

如果一份报告只回答“有哪些竞品、价格多少、卖点是什么”，它只能算资料整理；如果能回答“谁在什么场景下威胁我们、为什么、我们下一步如何应对”，才像竞品分析报告。

## 4. 推荐报告总结构

正式报告建议采用“商业分析正文 + 审计附录”结构。正文服务阅读和决策，附录服务追溯和答辩。

| 顺序 | 章节 | 写作目标 | 最低内容要求 |
|---:|---|---|---|
| 1 | 封面与报告信息 | 建立正式交付感 | 标题、目标产品、品类、数据范围、生成时间、版本、风险声明 |
| 2 | 执行摘要 | 第一屏给结论 | 一句话判断、最大威胁、最大机会、首要动作、证据等级 |
| 3 | 研究问题与分析范围 | 说明为什么分析 | 研究目标、竞品选择口径、数据来源、分析边界 |
| 4 | 类目与市场背景 | 解释竞争发生的环境 | 价格带、人群、场景、购买决策因素、类目关键矛盾 |
| 5 | 竞品选择与分层 | 说明为什么是这些竞品 | 直接竞品、替代竞品、低价威胁、高端标杆、排除对象 |
| 6 | 竞争格局判断 | 给出战场地图 | 按价格带、人群、场景拆解核心竞争关系 |
| 7 | 核心竞品 Battlecard | 把竞品变成可应对对象 | 竞品强项、弱项、用户比较理由、我方回应、风险边界 |
| 8 | 用户决策链分析 | 解释用户怎样被竞品影响 | 兴趣形成、能力理解、信任建立、成本评估、下单决策 |
| 9 | 差距矩阵 | 把差异转成问题清单 | 功能、价格、证据、表达、转化、售后/维护差距 |
| 10 | 机会地图与优先级 | 输出可执行动作 | P0/P1/P2 动作、责任方向、预期影响、所需证据 |
| 11 | 风险与证据边界 | 防止过度断言 | 可采纳结论、推断结论、暂无可靠数据、建议复核项 |
| 12 | 附录：证据、QA 与 Trace | 支撑可追溯演示 | Evidence 索引、Claim 绑定、QA 打回、补齐前后差异、Agent 流程摘要 |

其中第 2 至第 11 章是用户真正阅读的主体。第 12 章不应喧宾夺主，但必须保留，因为本项目的核心卖点包括可追溯、可打回、可人工修正。

## 5. 自动猫砂盆报告章节模板

### 5.1 封面与报告信息

封面应体现正式性，而不是技术演示感。

建议字段：

| 字段 | 写法 |
|---|---|
| 报告标题 | `自动猫砂盆竞品分析报告：<目标产品> 与核心竞品竞争关系重建` |
| 报告对象 | 目标 SKU 名称、品牌、类目 |
| 数据范围 | 用户提供的脱敏 SKU 快照、评论摘要、补充调研材料 |
| 数据时间 | 快照访问时间或“基于本地脱敏快照，非实时全网数据” |
| 报告版本 | v1.0 / 系统生成版本 |
| 风险声明 | 本报告不补写无证据价格、认证、销量、尺寸、排名；推断内容已显式标注 |

封面不展示 `task_id`、`product_id`、`evidence_id` 等内部字段。

### 5.2 执行摘要

执行摘要是最重要的章节。它必须让评委或产品经理在 30 秒内知道结论。

推荐结构：

| 模块 | 内容 |
|---|---|
| 一句话判断 | 当前目标产品最主要的竞争压力来自哪里 |
| 最大威胁 | 竞品名称、威胁场景、威胁原因 |
| 最大机会 | 目标产品最容易建立差异化的位置 |
| 首要动作 | 当前最应该优先处理的产品/内容/证据动作 |
| 决策可用性 | 可直接采纳 / 谨慎参考 / 证据不足 |
| 证据等级 | 高 / 中 / 低，并说明主要限制 |

写作句式：

```text
在本轮脱敏 SKU 快照范围内，<目标产品> 的核心竞争压力不是来自全量自动猫砂盆，而是来自 <价格带/人群/场景> 下同时满足 <关键能力> 与 <关键购买理由> 的 <竞品类型>。当前最应优先处理的是 <动作>，原因是 <证据支持的差距或风险>。
```

注意：

1. 不要把执行摘要写成“本报告分析了若干竞品”。
2. 不要在摘要里罗列所有 SKU。
3. 不要出现没有证据支持的市场排名或绝对化判断。

### 5.3 研究问题与分析范围

该章节用于证明报告不是随意拼接，而是有明确研究问题。

必须写清：

1. 本次分析要解决的主问题：例如“目标自动猫砂盆在中端价格带是否具备差异化竞争力”。
2. 竞品选择口径：同类自动猫砂盆、同价位替代方案、手动/半自动替代产品、高端标杆产品。
3. 数据来源：SKU 快照、商品页摘要、评论摘要、问卷/访谈文本、人工修正。
4. 数据边界：不是实时全网采集，不代表完整市场份额，不推断无证据事实。
5. QA 状态：是否发生证据补齐或打回。

推荐表格：

| 项目 | 本轮口径 |
|---|---|
| 目标类目 | 自动猫砂盆 |
| 目标用户 | 多猫家庭、上班族、重除臭需求、低维护成本敏感用户等 |
| 核心场景 | 清理负担、气味控制、宠物安全、耗材维护、智能提醒 |
| 价格切片 | 低价入门 / 中端主流 / 高端智能 |
| 数据来源 | 脱敏 SKU 快照 + 评论摘要 + 用户研究文本 |
| 不覆盖内容 | 实时销量、全网排名、未证实认证、医疗/安全效果绝对承诺 |

### 5.4 类目与市场背景

正式报告需要给读者一个“为什么这个类目这样竞争”的背景层。

自动猫砂盆场景建议围绕五个类目矛盾写：

1. 便利性与维护成本：自动清理是否真的减少日常负担，耗材和清洗是否反向增加成本。
2. 除臭能力与使用环境：小户型、多猫家庭、封闭空间更关注气味控制。
3. 智能功能与可靠性：App、称重、提醒、异常检测等功能是否稳定。
4. 宠物安全与信任建立：安全保护只能保守表达，必须有证据才可写具体认证或机制。
5. 价格带与转化阻力：高价产品需要更强证据证明长期价值，低价产品容易形成替代压力。

写法要点：

1. 背景只写与后文判断有关的内容。
2. 不要凭常识补市场规模、排名、增长率。
3. 如果缺少宏观数据，可以写“本轮快照无法支持市场规模判断，本章仅讨论样本内竞争结构”。

### 5.5 竞品选择与分层

参考样本中的成熟做法是先说明“为什么选这些竞品”，再展开分析。

推荐分层：

| 竞品层级 | 定义 | 自动猫砂盆示例口径 |
|---|---|---|
| 直接竞品 | 同类目、同任务、同价格带或相近价格带 | 其他自动猫砂盆 SKU |
| 替代竞品 | 不完全同类目，但解决同一用户任务 | 半自动猫砂盆、封闭式猫砂盆、除臭方案 |
| 低价威胁 | 价格显著更低但满足基础需求 | 入门自动款或手动替代方案 |
| 高端标杆 | 价格更高但提供信任、智能或品牌背书 | 高端智能自动猫砂盆 |
| 证据不足对象 | 样本中出现但证据无法支撑判断 | 缺价格时间、缺截图、缺评论摘要的 SKU |

每个入选竞品至少说明：

1. 入选原因。
2. 它在哪个切片下构成威胁。
3. 支撑证据是否充分。
4. 不纳入核心比较的原因，如证据不足或用户任务不一致。

### 5.6 竞争格局判断

竞争格局不是全量 SKU 列表，而是把关系压缩成“战场地图”。

建议至少输出三类切片：

| 切片 | 需要回答的问题 |
|---|---|
| 价格带 | 目标产品在低价、中端、高端分别会被谁替代？ |
| 人群 | 多猫家庭、预算敏感用户、智能功能偏好用户分别会比较谁？ |
| 场景 | 重除臭、低维护、宠物安全、远程看护等场景下谁更强？ |

推荐表达：

```text
在 <价格带/人群/场景> 切片下，<竞品 A> 的威胁高于 <竞品 B>。原因不是单一价格更低，而是它同时覆盖了 <购买理由 1> 和 <购买理由 2>；目标产品若要减少替代风险，应优先补强 <差距>。
```

每条竞争格局判断必须绑定 Claim 或 Evidence。若没有证据，写“暂无可靠数据”，不要为了完整性补猜。

### 5.7 核心竞品 Battlecard

Battlecard 是让报告变正式的关键。它把竞品从“资料卡”变成“可应对对象”。

每个核心竞品建议包含：

| 字段 | 写作要求 |
|---|---|
| 竞品定位 | 一句话说明该竞品被用户拿来比较的原因 |
| 威胁等级 | 高 / 中 / 低，必须说明对应切片 |
| 用户比较理由 | 用户为什么会把它和目标产品放在一起看 |
| 竞品强项 | 价格、功能、卖点表达、评论信号、品牌信任等 |
| 竞品弱项 | 维护成本、证据不足、功能边界、场景限制等 |
| 我方回应 | 目标产品应如何在产品、内容、证据或运营上回应 |
| 销售/页面话术 | 保守、可证据化的回应话术 |
| 风险边界 | 哪些结论只是推断，哪些暂无可靠数据 |

不要只写：

```text
竞品 A 价格为 X，卖点为 Y，评论为 Z。
```

应该写：

```text
竞品 A 在预算敏感用户中构成直接压力，因为它用更低进入门槛覆盖了“自动清理 + 基础除臭”的核心任务。目标产品不宜只强调功能数量，而应把“长期维护省心”和“关键安全机制证据”前置，否则用户会把价格差理解为非必要溢价。
```

### 5.8 用户决策链分析

产品经理类参考报告通常会把分析落到用户路径。本项目应把自动猫砂盆购买决策拆成五段：

| 决策阶段 | 用户问题 | 报告应分析的竞争点 |
|---|---|---|
| 信息触达 | 我为什么需要自动猫砂盆？ | 痛点表达、场景图片、评论共现 |
| 兴趣形成 | 它能解决我的哪类麻烦？ | 自动清理、除臭、低维护、多猫适配 |
| 能力理解 | 它和别的产品差在哪？ | 功能树、价格带、耗材/维护成本 |
| 信任建立 | 是否安全、稳定、售后可靠？ | 安全证据、评论风险、认证/保障信息 |
| 下单决策 | 我为什么现在买它？ | 价格、优惠、套装、风险承诺、页面转化 |

输出方式建议用表格，而不是长段：

| 阶段 | 竞品影响 | 目标产品风险 | 建议动作 | 证据状态 |
|---|---|---|---|---|
| 信任建立 | 竞品强调安全机制 | 目标产品若缺截图或认证证据，结论只能谨慎参考 | 补齐安全机制页面证据，页面表达改为保守说明 | 需复核 |

### 5.9 差距矩阵

差距矩阵用于把“看起来差不多”拆成可行动的问题。

建议维度：

| 维度 | 比较内容 | 输出形式 |
|---|---|---|
| 功能能力差距 | 清洁、除臭、容量、安全、智能、维护 | 目标状态、竞品参照、影响 |
| 价格与成本差距 | 标价、到手价、耗材、维护成本 | 是否影响转化 |
| 证据差距 | 价格时间、截图、评论摘要、认证材料 | 是否需要补采 |
| 卖点表达差距 | 是否把功能转成用户收益 | 页面/报告表达建议 |
| 评论信号差距 | 痛点、购买理由、异议、信任因素 | 关联用户决策阶段 |
| 转化差距 | 优惠、套装、售后、风险承诺 | 运营动作 |

每条差距必须回答：

1. 差在哪里？
2. 对用户决策有什么影响？
3. 下一步动作是什么？
4. 证据够不够？

### 5.10 机会地图与优先级

正式报告不能只写“建议关注”。每条建议至少包含动作、责任方向、优先级、预期影响和证据边界。

推荐字段：

| 字段 | 示例 |
|---|---|
| 优先级 | P0 / P1 / P2 |
| 动作类型 | 产品 / 定价 / 内容 / 证据 / 评论 / 运营 |
| 动作说明 | 补齐关键安全机制截图；重写除臭卖点为用户收益；增加维护成本说明 |
| 责任方向 | 产品、运营、内容、数据采集、人工复核 |
| 目标切片 | 多猫家庭 / 重除臭 / 预算敏感 / 中端价格带 |
| 预期影响 | 降低替代风险、提升信任建立、减少价格异议 |
| 证据边界 | 现有证据支持 / 推断 / 暂无可靠数据 |

推荐排序逻辑：

```text
优先级 = 用户决策影响 × 竞品威胁 × 证据可信度 ÷ 执行成本
```

规则：

1. P0 动作不能来自完全无证据结论。
2. 证据不足的动作可以是“补证据”，不能直接是“调整产品定位”。
3. 涉及宠物安全、电器安全、认证等内容时，动作应偏向“补齐证据与保守表达”，不要写成绝对承诺。

## 6. 正式写作风格规则

### 6.1 段落结构

每个正文段落建议采用“判断 - 依据 - 影响 - 动作”结构：

1. 判断：先给结论。
2. 依据：说明来自哪些证据或信号。
3. 影响：解释对用户决策、竞品威胁或产品策略的意义。
4. 动作：给出下一步建议或复核要求。

示例骨架：

```text
<竞品/差距> 在 <切片> 下构成 <高/中/低> 威胁。支撑依据是 <证据摘要>，但 <证据限制>。这意味着目标产品在 <决策阶段> 可能面临 <影响>。建议 <责任方向> 优先 <动作>。
```

### 6.2 用词要求

推荐用词：

1. “在本轮样本范围内”
2. “证据显示”
3. “可判断为”
4. “更可能影响”
5. “建议复核”
6. “暂无可靠数据”
7. “该结论属于推断”
8. “当前证据不足以支持绝对判断”

避免用词：

1. “全网第一”
2. “绝对领先”
3. “必然胜出”
4. “安全无风险”
5. “销量最高”
6. “认证齐全”
7. “用户都认为”
8. “市场已经证明”

除非有可靠证据，否则不得写价格、销量、认证、尺寸、排名、市场份额等具体事实。

### 6.3 内容去重

报告正文要减少重复搬运同一 SKU 信息：

1. SKU 基础字段只在首次出现时简要说明。
2. 价格、评分、评论数只在影响判断时出现。
3. 相同 Evidence 摘要在同一正文章节最多展开一次。
4. 完整证据列表放入附录，不在正文反复铺开。
5. Trace、Agent、工具调用、Token 等系统信息只进入附录或过程追踪页。

### 6.4 正式程度

正式报告应避免口语化和流水账。

不推荐：

```text
这个竞品价格比较低，卖点也还可以，所以我们可以关注一下。
```

推荐：

```text
该竞品在入门价格带具备较强替代压力，原因是其基础自动清理能力已经覆盖预算敏感用户的核心任务。目标产品若继续主打功能数量，可能难以解释价格差异。建议优先补强“长期维护省心”的证据表达，并将安全机制说明前置到详情页核心区域。
```

## 7. 内容丰富度要求

为避免报告“内容太少”，Writer Agent 生成报告时应满足以下最低丰富度：

| 模块 | 最低要求 |
|---|---:|
| 执行摘要 | 至少 5 个结构化要点 |
| 竞品选择与分层 | 至少覆盖 3 类竞品层级 |
| 竞争格局判断 | 至少 3 个切片判断 |
| 核心 Battlecard | 至少 3 个核心竞品；如样本不足，说明不足原因 |
| 用户决策链 | 至少覆盖 5 个决策阶段 |
| 差距矩阵 | 至少 6 条差距；覆盖不少于 3 类维度 |
| 机会地图 | 至少 3 条机会动作，含 P0/P1/P2 |
| 风险与证据边界 | 至少列出可采纳、推断、暂无可靠数据三类 |
| 附录 | Evidence 索引、QA 打回、Trace 摘要必须保留 |

如果数据不足以满足数量要求，不允许编造内容；应生成“样本不足说明”和“补采建议”。

## 8. 证据与合规规则

### 8.1 证据等级

建议报告中使用三级证据等级：

| 等级 | 条件 | 报告写法 |
|---|---|---|
| 高 | 有来源、访问时间、截图或稳定快照，且多条证据一致 | 可作为主要结论 |
| 中 | 有来源和摘要，但缺部分时效字段或截图 | 谨慎参考，并说明限制 |
| 低 | 只有单条评论、单条摘要或人工推断 | 只能作为方向线索 |

### 8.2 事实、推断和暂无可靠数据

三类内容必须区分：

| 类型 | 定义 | 写法 |
|---|---|---|
| 事实 | 快照或材料直接支持 | “证据显示……” |
| 推断 | 基于多个信号归纳，但非直接事实 | “推断：……” |
| 暂无可靠数据 | 缺证据或证据冲突 | “暂无可靠数据，不建议写入正式结论” |

### 8.3 宠物安全与电器安全

自动猫砂盆涉及宠物安全和电器安全，表达必须保守：

1. 不能写“绝对安全”“完全无风险”。
2. 未见可靠材料时，不得补写认证、检测、保护机制。
3. 如果评论中出现安全担忧，只能作为评论信号，不得放大成事实结论。
4. 建议动作应写“补齐安全机制证据”“保守说明适用边界”“建议人工复核”。

## 9. 对 Writer Agent 的落地要求

Writer Agent 不应直接把 Product、Evidence、CompetitionEdge 翻译成段落，而应优先消费分析型 Artifact：

1. `StrategyBrief`：用于执行摘要和竞争格局总判断。
2. `CompetitorBattlecard`：用于核心竞品章节。
3. `GapMatrix`：用于差距矩阵章节。
4. `OpportunityMap`：用于机会地图与优先级章节。
5. `ReviewSignalCluster`：用于用户决策链和评论信号分析。
6. `EvidenceQualitySummary`：用于风险与证据边界。
7. `TraceSummary`：用于附录和答辩支撑。

如果某些 Artifact 暂未实现，Writer Agent 应用规则兜底生成，但仍要遵守：

1. 正文先结论，后证据，最后动作。
2. 每条核心判断必须能追溯 Claim 或 Evidence。
3. 每条建议必须含优先级、责任方向和证据边界。
4. 不把内部 ID 暴露在正文。
5. 不把 QA、Trace、Token 当作正文主线。
6. 未配置模型 API Key 时，规则流程仍应生成完整报告。

## 10. 报告质量检查清单

报告生成完成后，应至少检查：

1. 首页是否能回答“谁威胁最大、为什么、先做什么”。
2. 是否说明竞品选择标准，而不是直接罗列竞品。
3. 是否至少有 3 个核心竞品 Battlecard。
4. 是否至少有 1 张差距矩阵。
5. 是否至少有 3 条机会动作，且带 P0/P1/P2。
6. 每条核心事实是否绑定 Evidence 或 Claim。
7. 推断是否显式标记。
8. 证据不足是否写“暂无可靠数据”或“建议复核”。
9. 正文是否避免重复铺 SKU 字段。
10. 是否没有泄露 API Key、账号、手机号、未脱敏隐私。
11. 是否没有在正文暴露 `task_id`、`claim_id`、`evidence_id`、`product_id` 等内部字段。
12. QA 打回和补齐前后差异是否在 Trace 或附录保留。
13. Word 导出和网页报告章节是否一致。
14. 报告表达是否正式、可汇报、可打印。

## 12. 后续实施建议

建议后续报告升级按以下顺序推进：

1. 先让 Analysis Agent 产出 `CompetitorBattlecard`、`GapMatrix` 和 `OpportunityMap`。
2. 再让 Writer Agent 按本文档章节组织正式报告。
3. 增加报告质量规则，检查“无商业含义、无动作建议、重复 SKU 字段、内部 ID 泄露、证据越界”。
4. 前端报告页优先展示执行摘要、Battlecard、差距矩阵和机会地图。
5. Word 导出保持与网页报告相同章节，并把 Evidence、QA、Trace 放入附录。

这样升级后，报告会更接近参考目录中的正式竞品分析报告：有背景、有对象选择、有矩阵、有判断、有行动、有风险边界，也保留本项目独有的证据链与 QA 打回能力。

## 13. 基于本规范的项目改进点与详细更新计划

### 13.1 当前项目基线

截至 2026-06-09，项目已经具备一轮报告智能化升级基础：

1. Analysis Agent 已能产出 `StrategyBrief`、`CompetitorBattlecard`、`GapMatrixItem`、`OpportunityItem` 等商业分析 Artifact。
2. Writer Agent 已能优先消费分析 Artifact，并生成 `narrative_report` 正式报告正文。
3. 网页报告和 Word 导出已支持 Battlecard、差距矩阵、机会地图等新版内容的基础展示。

5. Word `.docx` 已替代 Markdown 作为正式导出入口，Markdown 仅可作为内部规范或测试对照。

### 13.3 总体升级目标

本轮升级目标是把报告从“已有正式叙事层”进一步提升为“结构完整、内容充实、证据边界清楚、可直接汇报的竞品分析报告”。

验收时应满足：

1. 网页报告和 Word 报告都能覆盖本文档第 4 章的正式章节结构。
2. 执行摘要第一屏能回答“谁威胁最大、为什么、先做什么”。
3. 报告必须说明研究问题、数据范围和竞品选择口径。
4. 至少输出 3 个核心竞品 Battlecard；样本不足时必须说明原因。
5. 至少输出 6 条差距矩阵项，并覆盖不少于 3 类差距维度。
6. 至少输出 3 条机会动作，包含 P0/P1/P2、责任方向、预期影响和证据边界。
7. 用户决策链必须覆盖信息触达、兴趣形成、能力理解、信任建立、下单决策 5 个阶段。
8. 每条核心判断能追溯 Evidence 或 Claim。
9. 推断内容显式标注；证据不足写“暂无可靠数据”或“建议复核”。
10. 未配置模型 API Key 时，本地规则流程仍可生成完整报告。

### 13.4 阶段 0：冻结报告升级验收口径

目标：先把“什么叫正式且内容够多”变成测试可检查的标准。

改进内容：

1. 将本文档中的章节结构、内容丰富度和证据边界同步为后端 Writer 与前端报告页的验收口径。
2. 增加一份报告期望骨架 fixture，用于测试章节、字段和内容数量，不使用模拟数据替代最终 Demo 数据。
3. 明确本轮升级仍只使用本地脱敏 SKU 快照和用户研究文本，不做真实外部采集。

### 13.5 阶段 1：增强分析 Artifact Schema

目标：让 Analysis Agent 产出的结构足够支撑正式报告，而不是依赖 Writer 临时补脑。

改进内容：

1. 扩展 `StrategyBrief`：
   - 增加 `research_question`：本轮分析要回答的主问题。
   - 增加 `analysis_scope`：数据范围、切片范围、样本数量。
   - 增加 `category_tensions`：自动猫砂盆类目的关键竞争矛盾，例如除臭、维护成本、安全信任、智能可靠性。
   - 增加 `competitor_selection_rationale`：竞品选择口径。
2. 扩展 `CompetitorBattlecard`：
   - 增加 `competitor_tier`：direct / substitute / low_price_threat / high_end_benchmark / evidence_insufficient。
   - 增加 `threat_level`：high / medium / low。
   - 增加 `target_slice`：价格带、人群、场景。
   - 增加 `evidence_status`：high / medium / low / missing。
   - 增加 `do_not_overclaim`：该竞品分析中不能过度断言的内容。
3. 扩展 `GapMatrixItem`：
   - 增加 `gap_type`：function / price_cost / evidence / messaging / review_signal / conversion / service。
   - 增加 `evidence_status`。
   - 增加 `next_step_owner`。
4. 扩展 `OpportunityItem`：
   - 增加 `action_type`：product / pricing / content / evidence / review / positioning / operation。
   - 增加 `acceptance_signal`：动作完成后如何判断有效。
   - 增加 `must_not_claim`：证据不足时不能写入报告的内容。

建议修改文件：

| 文件 | 修改点 |
|---|---|
| `backend/app/schemas/analysis_artifact.py` | 扩展 Pydantic Schema |
| `backend/app/agents/analysis.py` | 生成新增字段 |
| `backend/app/graph/state.py` | 如新增 Artifact 类型，纳入 state |
| `backend/tests/test_analysis_agent.py` | 增加字段生成与证据边界测试 |
| `backend/tests/test_graph_state.py` | 验证新增 Artifact 可序列化 |

测试与验收：

1. Schema 合法样例校验通过。
2. Analysis Agent 至少生成 1 个 `StrategyBrief`，且包含研究问题、范围和竞品选择口径。
3. 每个核心 Battlecard 都有竞品层级、威胁等级、目标切片和证据状态。
4. 缺证据时 `evidence_status=missing` 或 `low`，且 `do_not_overclaim` 不为空。
5. 不引入新数据库表，继续使用 Artifact JSON 存储。

### 13.6 阶段 2：新增 ReviewSignalCluster 与类目背景生成

目标：让报告拥有“用户为什么比较、为什么犹豫”的分析深度。

改进内容：

1. 新增 `ReviewSignalCluster` Schema：
   - `signal_type`：pain / buying_reason / objection / trust_factor / maintenance_cost / safety_concern。
   - `signal_summary`：用户信号摘要。
   - `affected_products`：影响哪些产品。
   - `related_decision_stage`：对应用户决策阶段。
   - `evidence_ids`：来源证据。
   - `action_hint`：对产品或页面表达的动作提示。
   - `evidence_status`：证据强度。
2. 在 Collection 或 Analysis 阶段将现有 `ReviewInsight`、评论摘要和 `research_text` 归一为 ReviewSignalCluster。
3. 新增类目背景生成规则：
   - 自动猫砂盆固定类目矛盾：便利性 vs 维护成本、除臭能力 vs 小户型环境、智能功能 vs 可靠性、宠物安全 vs 证据边界、价格带 vs 转化阻力。
   - 只使用规则和已有证据，不补写市场规模或实时排名。
4. 将 ReviewSignalCluster 供 Writer 的用户决策链、差距矩阵和机会地图使用。

建议修改文件：

| 文件 | 修改点 |
|---|---|
| `backend/app/schemas/analysis_artifact.py` | 新增 `ReviewSignalCluster` |
| `backend/app/agents/analysis.py` 或新增轻量服务 | 从评论和调研文本生成 signal cluster |
| `backend/app/services/task_execution.py` | 缓存新增 Artifact |
| `backend/app/services/trace_service.py` | Trace 可展示 signal cluster 摘要 |
| `backend/tests/test_analysis_agent.py` | 覆盖评论信号归一 |

测试与验收：

1. 至少生成 pain、buying_reason、objection、trust_factor 中的 2 类信号。
2. 每个 signal 必须有 evidence_ids 或明确 `evidence_status=missing`。
3. 用户决策链章节能消费 ReviewSignalCluster，而不是只消费 CompetitionEdge。
4. 没有评论数据时，报告写“暂无可靠评论信号”，并给出补采建议。

### 13.7 阶段 3：升级 Report Planner 为完整正式章节

目标：让报告结构从 5 个叙事章节升级为完整正式报告骨架。

改进内容：

1. 将 `narrative_report.sections` 扩展为以下稳定章节：
   - `report_info`
   - `executive_summary`
   - `research_question_and_scope`
   - `category_context`
   - `competitor_selection`
   - `competitive_landscape`
   - `core_competitor_battlecards`
   - `decision_chain`
   - `gap_matrix`
   - `opportunity_map`
   - `risk_and_evidence_boundary`
   - `appendix_traceability`
2. Report Planner 按章节消费不同 Artifact：
   - `report_info` 消费 task、target product、data source metadata。
   - `research_question_and_scope` 消费 StrategyBrief。
   - `competitor_selection` 消费 Battlecard 的 tier 和 evidence_status。
   - `decision_chain` 消费 ReviewSignalCluster 和 CompetitionEdge。
   - `gap_matrix` 消费 GapMatrixItem。
   - `opportunity_map` 消费 OpportunityItem。
   - `risk_and_evidence_boundary` 消费 risk flags、QA、ReportQualityCheck。
3. 增加章节最小内容量：
   - Battlecard 不少于 3 个，样本不足必须说明。
   - GapMatrix 不少于 6 条，覆盖不少于 3 类 gap_type。
   - Opportunity 不少于 3 条，必须含 P0/P1/P2。
4. 保留旧 `ReportData` 八章节外壳兼容，但网页和 Word 优先渲染新版 `narrative_report`。

建议修改文件：

| 文件 | 修改点 |
|---|---|
| `backend/app/agents/writer.py` | 重构 `_build_narrative_report_plan` 和 fallback sections |
| `backend/app/schemas/report.py` | 如有必要，收紧 `narrative_report` 结构文档或类型 |
| `backend/tests/test_writer_agent.py` | 增加完整章节顺序和数量断言 |
| `frontend/src/api/schema.ts` | 同步 OpenAPI 类型后更新 |

测试与验收：

1. `narrative_report.sections` 包含完整章节 ID，顺序稳定。
2. 无 LLM 时 fallback 也能生成完整章节。
3. 正文不出现内部 ID。
4. 证据不足章节不会空白，而是给出“暂无可靠数据/建议补采”。

### 13.8 阶段 4：升级 Writer LLM Prompt 与本地规则兜底

目标：让 LLM 和本地规则都按照正式报告口径写，不再输出泛泛短文。

改进内容：

1. 更新 `writer_formal_narrative_report_generation`：
   - 明确只允许输出完整正式章节。
   - 每节必须“先判断、再依据、再影响、最后动作”。
   - 不得补写价格、销量、认证、尺寸、排名。
   - 证据不足必须写“暂无可靠数据”或“建议复核”。
2. 按章节拆分 LLM 生成或保留总编辑层，但输入必须含：
   - research question。
   - competitor tier。
   - Battlecard。
   - ReviewSignalCluster。
   - GapMatrix。
   - OpportunityMap。
   - QA 与证据边界。
3. 增强本地 fallback：
   - 用固定模板生成报告信息、研究范围、类目背景、竞品分层。
   - 用 Artifact 数据生成表格化 Battlecard、GapMatrix、OpportunityMap。
   - 样本不足时输出补采建议。

建议修改文件：

| 文件 | 修改点 |
|---|---|
| `backend/app/agents/writer.py` | Prompt、fallback、章节生成、metadata |
| `backend/tests/test_writer_agent.py` | LLM 调用顺序、fallback 完整性、证据保守性 |

测试与验收：

1. Fake LLM 输出时，报告章节和字段被正确应用。
2. LLM fallback 时，报告仍有完整章节和最低内容量。
3. LLM 返回过度断言时，质量规则或修复流程能降级。
4. Token usage 和 metadata 能区分章节生成、质检和修复。

### 13.9 阶段 5：强化报告质量规则

| no_unsupported_market_claim | 是否出现无证据市场规模、排名、认证、销量 | 标记 needs_revision |
| safety_conservative_language | 宠物安全、电器安全表达是否保守 | 标记 needs_revision |
