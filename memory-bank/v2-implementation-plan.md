# 竞品分析系统 2.0 实施计划

## 1. 使用说明

本文档位于 `memory-bank/`，基于 `memory-bank/v2-modification-notes.md`，并结合现有代码、`memory-bank/architecture.md`、`memory-bank/design-document.md`、`memory-bank/implementation-plan.md`、`memory-bank/progress.md`、`memory-bank/tech-stack.md` 编写，用于指导 AI 开发者按小步迭代完成 2.0 改版。

当前 1.0 基线已经完成：FastAPI 后端、SQLite 存储、LangGraph DAG、Collection/Analysis/QA/Writer 四 Agent、QA 真实打回、Trace、Human Review、竞争图谱、产品画像、网页报告、Markdown 导出、Playwright Demo 路径和 Demo 冻结。2.0 不是推翻重做，而是在现有能力上把默认体验从“AI 产物陈列”升级为“面向产品经理的竞品决策工作台”，并把 Markdown 交付升级为 Word `.docx` 交付。

执行规则：

1. 每一步都必须小而具体，不跨多个责任边界。
2. 每一步都必须包含验证正确性的测试。
3. 每一步完成后必须运行对应测试，并把通过或失败结果记录到 `memory-bank/progress.md`。
4. 每完成一个重大功能或里程碑，必须更新 `memory-bank/architecture.md`。
5. 严禁跳过 LangGraph、四 Agent、QA 打回、证据链、Trace、Human Review、报告导出这些核心能力。
6. 严禁引入 Celery、Redis、PostgreSQL、Next.js、Redux、Tailwind、复杂实时采集平台、后端 PDF 服务、Headless Office、Graphviz 系统依赖或微服务架构。
7. 允许新增轻量后端依赖 `python-docx` 与 `Pillow`，仅用于 Word 报告和简化关系图生成。
8. 未配置模型 API Key 时，系统必须仍能依靠本地快照和规则流程跑通完整 Demo。
9. 不得在无证据情况下补价格、认证、尺寸、销量、排名；找不到可靠证据时写“暂无可靠数据”。
10. 本计划只写实施指令，不包含任何代码片段。
11. 2.0 默认实现不得把模型调用作为必需链路；但必须保留可选模型增强扩展点，后续真实场景可在配置 API Key 后用于解释生成或表达润色。模型输出必须经过结构化解析、Pydantic 校验、证据约束和 QA 检查；模型不得新增事实，与结构化证据冲突时以证据和 QA 结果为准。

## 1.1 已确认补充决策

以下决策用于消除 2.0 实施中的歧义：

1. Markdown 导出接口在 2.0 中直接删除，不保留用户可见入口，也不保留兼容路由；对应 OpenAPI、前端调用和回归测试必须同步删除或更新。
2. Word 导出成功时 `GET /tasks/{task_id}/report/docx` 直接返回 `.docx` 文件下载响应；失败时返回标准错误提示，并且不得影响网页报告阅读。
3. 产品图片字段必须返回浏览器可访问 URL，不向前端暴露本地绝对文件路径。
4. 产品主图优先使用快照或素材中记录的远程图片 URL；远程 URL 不可访问时，再回退到本地截图或 Evidence 截图对应的可访问 URL；均不可用时返回“暂无可靠图片”状态。
5. 判断强度、决策可用状态、证据可信状态、威胁等级和行动建议优先级必须使用固定标准，并在接口数据中携带对应原因，供前端展示在状态旁边。
6. 关键结论缺少下钻引用时，生成服务优先标记风险并给出原因；Schema 只拦截完全不可展示或结构不完整的数据。
7. PM 关系标签采用规则化启发式映射，从竞争类型、评分维度、证据状态、决策阶段、卖点和评论信号中推导，不逐个 SKU 硬编码。
8. “对策略动作最有启发的竞品”按“有较高竞争分、证据至少基本可用、能生成明确行动建议”的关系优先筛选。
9. 证据与过程追踪的下钻入口先通过 URL query 参数定位到目标 Tab，并在可获得具体对象 ID 时高亮对应条目。
10. Word 目录采用静态章节目录即可，不要求生成需要 Office 刷新域的自动目录。
11. 步骤 01 的迁移清单固定落在 `memory-bank/v2-migration-checklist.md`。

## 1.2 2.0 状态与分数标准

后端生成 2.0 展示状态时必须同时返回状态值和原因说明，前端在状态旁边展示简短原因。

### 判断强度

| 等级 | 固定标准 |
|---|---|
| 明确判断 | 核心结论证据完整或基本完整，关键 Evidence 可追溯，平均置信度不低于 0.75，且不存在未解决的高严重度 QA 风险 |
| 倾向判断 | 证据支持方向但存在局限，平均置信度不低于 0.55，或存在已标注但不阻断的中低风险 |
| 仅作假设 | 置信度低于 0.55，或核心证据缺失、关键结论只能作为后续验证方向 |

### 证据可信状态

| 状态 | 固定标准 |
|---|---|
| 可直接采纳 | 关键 Evidence 有来源、访问时间或可解释的快照时间、内容摘要和局限性，且无未解决关键 QA 风险 |
| 谨慎参考 | Evidence 基本可追溯，但存在单项缺失、低可信来源、时间局限或非阻断 QA 风险 |
| 证据不足 | 缺少关键 Evidence、缺少关键时效字段且未修复，或存在未解决高严重度 QA 风险 |

### 决策可用状态

| 状态 | 固定标准 |
|---|---|
| 可用于初步决策 | 关键结论均为“明确判断”或“倾向判断”，证据可信状态为“可直接采纳”或“谨慎参考”，且 QA 风险已解决或可接受 |
| 建议谨慎决策 | 存在低可信证据、局部缺失或未完全解决的中低风险，但仍可作为产品讨论输入 |
| 仅供方向参考 | 核心结论证据不足，或存在未解决高严重度 QA 风险，不建议直接用于产品决策 |

### 威胁等级

| 等级 | 固定标准 |
|---|---|
| 高威胁 | 竞争分大于等于 0.80，且证据可信状态不是“证据不足” |
| 中威胁 | 竞争分大于等于 0.60 且小于 0.80，且证据至少可谨慎参考 |
| 低威胁 | 竞争分低于 0.60，或竞争关系只在弱切片中成立 |
| 高分需复核 | 竞争分大于等于 0.60，但证据不足、关键字段缺失或存在未解决 QA 风险 |

### 行动建议优先级

| 优先级 | 固定标准 |
|---|---|
| P0 立即处理 | 与高威胁关系、关键转化阻碍、合规风险或证据补齐阻断项直接相关 |
| P1 本轮优化 | 与中威胁关系、明确表达机会、可落地产品优化或可验证价格策略相关 |
| P2 后续验证 | 主要来自倾向判断或假设，需要更多证据、调研或后续版本验证 |

## 2. 当前基线与迁移边界

AI 开发者开始任何 2.0 开发前，必须确认以下现状：

1. `data/snapshots/demo_sku_snapshot.json` 是当前正式 Demo 快照输入。
2. `demo/stable-demo-input.json` 是答辩和录屏稳定输入。
3. 默认目标为 `sku_02`，QA 打回样例为 `sku_01` 缺失 `source.access_time`。
4. `GET /tasks/{task_id}/report/markdown` 当前存在，但 2.0 必须直接删除该接口、移除前端可见入口，并新增 Word 导出。
5. 创建任务后当前默认跳转到过程追踪页；2.0 必须改为跳转到竞争态势总览页。
6. 当前前端主要集中在 `frontend/src/App.tsx`；2.0 可以在必要时拆分组件，但不得为了重构而重构。
7. 当前 OpenAPI 类型通过 `scripts/sync-openapi-types.mjs` 同步到 `frontend/src/api/schema.ts`；后端接口变更后必须同步类型。

## 3. 里程碑顺序

推荐按以下里程碑推进：

1. 后端 2.0 契约与总览 API。
2. 图片路径、关键关系筛选与竞争图谱解释层。
3. 报告数据结构、Word 导出与简化关系图。
4. 证据与过程追踪数据重组。
5. 前端导航、总览页和下钻路径。
6. 前端图谱、画像、报告和过程追踪改版。
7. 全站中文化、响应式、E2E 和 Demo 冻结更新。

## 4. 分步实施指令

### 步骤 01：建立 2.0 契约清单

执行指令：

1. 梳理现有后端 Schema、API、前端路由、测试文件与 2.0 PRD 的差异。
2. 在 `memory-bank/v2-migration-checklist.md` 形成迁移清单，明确哪些能力新增、哪些能力保留、哪些能力废弃。
3. 明确 Markdown 导出只允许作为历史能力被 Word 导出替换，2.0 中必须删除旧 Markdown API，不再作为用户可见交付入口。
4. 明确 `overview`、`battlefield`、`profile`、`report`、`trace` 的 2.0 责任边界。

验证测试：

1. 文档检查确认迁移清单覆盖总览、图谱、画像、报告、证据与过程追踪、DOCX、中文化。
2. 文档检查确认没有把非目标能力列入实施范围。
3. 文档检查确认每个后续步骤都有明确验证方式。

### 步骤 02：补充后端 2.0 枚举与展示状态 Schema

执行指令：

1. 在现有 Pydantic Schema 基础上补充判断强度、决策可用状态、证据可信状态、威胁等级、PM 关系标签、行动建议优先级和责任类型。
2. 所有新增枚举必须服务于 PM 可读表达，不替代底层技术枚举。
3. 保留原有 `Claim`、`Evidence`、`CompetitionEdge`、`ReviewTask` 等结构化对象，不破坏现有 Agent 传递协议。

验证测试：

1. 后端 Schema 测试确认新增枚举只接受允许值。
2. OpenAPI 测试确认新增 Schema 能被导出。
3. 回归测试确认现有核心 Schema 测试仍通过。

### 步骤 03：为 Product 增加主图路径派生能力

执行指令：

1. 为产品展示增加 `primary_image_path` 或等价主图 URL 数据来源，优先使用快照或素材中记录的远程图片 URL，其次使用 `data/raw/sku_xx` 首图，再其次使用 Evidence 截图。
2. 后端必须返回浏览器可访问 URL；远程 URL 不可访问时回退到本地截图对应的可访问 URL，图片缺失时统一返回“暂无可靠图片”的可展示状态，不让前端猜测。
3. 不修改冻结 Demo 的业务含义；如必须调整快照字段，必须同步 Demo 冻结记录和回归测试。

验证测试：

1. Snapshot Loader 测试确认每个可用 SKU 都能派生可访问主图 URL 或明确缺失状态。
2. 后端 Schema 测试确认 `primary_image_path` 缺失不会导致产品校验失败。
3. Demo 冻结测试确认默认目标和 QA fixture 不因图片字段变化而漂移。

### 步骤 04：实现分析范围汇总服务

执行指令：

1. 基于任务、快照、Product、Evidence 汇总 SKU 数量、数据来源、品类、子类、快照日期或访问时间范围。
2. 明确输出“本报告基于用户提供的脱敏 SKU 快照，不代表实时全网数据”。
3. 字段缺失时输出“暂无可靠数据”，不得推断实时数据或全网覆盖。

验证测试：

1. 后端单元测试确认分析范围能从冻结 Demo 快照得到 SKU 数量和来源说明。
2. 后端单元测试确认缺少访问时间时显示“暂无可靠数据”。
3. 安全测试确认分析范围输出不包含本地绝对隐私路径或 API Key。

### 步骤 05：实现总览数据结构

执行指令：

1. 定义 `OverviewData` 及其子结构，覆盖一句话判断、判断强度、决策可用状态、状态原因、分析范围、关键竞品、机会点、风险点、行动建议、当前切片和下钻引用。
2. 关键竞品默认包含最高威胁直接竞品、最高威胁替代竞品和需复核高分竞品；不存在的类别不得硬补。
3. 行动建议必须包含优先级和责任类型。
4. 每个关键结论都必须携带 Evidence 或 Trace 下钻引用；确实缺少引用时必须标记风险并说明缺失原因。

验证测试：

1. Schema 测试确认 `OverviewData` 合法样例可通过校验。
2. Schema 测试确认行动建议缺少优先级或责任类型时校验失败。
3. Schema 测试确认关键结论缺少下钻引用时被标记为风险，并确认完全不可展示的数据会校验失败。

### 步骤 06：实现总览服务

执行指令：

1. 基于现有 Workflow 产物生成 PM 可读总览，不让前端从 Profile、Battlefield、Report 拼接。
2. 一句话判断必须先给结论，再补证据风险状态。
3. 决策可用状态必须由 QA 风险、证据完整性和关键结论可信状态共同决定。
4. 切片参数变化后，总览中的判断、关键竞品、机会点、风险点和建议必须同步变化。
5. 总览服务必须按“2.0 状态与分数标准”输出固定状态、分数依据和原因说明。

验证测试：

1. 后端单元测试确认冻结 Demo 能生成总览。
2. 后端单元测试确认存在未解决 QA 风险时不会返回“可用于初步决策”。
3. 后端单元测试确认切换价格带或人群后关键竞品排序可能变化。
4. 后端单元测试确认总览没有裸露 `Product`、`Claim`、`Evidence` 字段名作为主文案。

### 步骤 07：新增总览 API

执行指令：

1. 新增 `GET /tasks/{task_id}/overview`。
2. 接口支持价格带、人群、使用场景切片参数。
3. 接口只在任务完成或人工复核状态下返回总览数据；未完成时返回标准错误。
4. 统一使用现有 `ApiResponse` 响应外壳。

验证测试：

1. API 测试确认完成任务可以获取总览数据。
2. API 测试确认未完成任务请求总览返回可理解错误。
3. API 测试确认切片参数会传递到总览服务。
4. API 测试确认不存在任务返回标准错误。

### 步骤 08：升级竞争图谱后端契约

执行指令：

1. 在现有 `BattlefieldData` 基础上补充关键关系筛选结果、入选理由、威胁等级、PM 关系标签、标签解释、四段式解释、应对建议、证据可信状态和产品图片路径。
2. 保留现有图节点、图边、评分解释、决策链、证据卡片和 QA 摘要。
3. 竞争分继续保留在下钻信息中，默认展示层弱化裸分数。

验证测试：

1. Schema 测试确认扩展后的 `BattlefieldData` 兼容现有字段。
2. API 合约测试确认每条关键关系包含入选理由和威胁等级。
3. API 合约测试确认每条关键关系包含四段式解释。

### 步骤 09：实现关键竞争关系筛选

执行指令：

1. 后端负责筛选默认展示的 3 到 5 条关键竞争关系。
2. 筛选策略必须覆盖最高威胁直接竞品、最高威胁替代竞品、需复核高分竞品和对策略动作最有启发的竞品。
3. “对策略动作最有启发的竞品”优先选择竞争分较高、证据至少基本可用、且能生成明确行动建议的关系。
4. 如果某类关系不存在，不强行补齐。
5. 提供“展开全部关系”所需的数据标记。

验证测试：

1. 后端单元测试确认默认关系数量在 3 到 5 条范围内，除非数据本身不足。
2. 后端单元测试确认高分但证据不足的边被标为需复核，而不是高威胁。
3. 后端单元测试确认展开全部时可以返回完整关系集合。
4. 回归测试确认原有切片过滤仍可用。

### 步骤 10：实现 PM 关系标签与威胁等级规则

执行指令：

1. 将底层竞争类型和评分信息映射为正面硬碰、低价截流、场景替代、信任压制、内容种草竞争等 PM 关系标签。
2. 每个标签必须提供一句简短解释。
3. 威胁等级必须考虑竞争分和证据可信状态；证据不足时不能直接标为高威胁。

验证测试：

1. 后端单元测试覆盖每一种 PM 关系标签。
2. 后端单元测试确认证据低可信时高分边显示为高分需复核。
3. 后端单元测试确认标签解释不是空文本。

### 步骤 11：实现竞争边四段式解释

执行指令：

1. 每条关键竞争边必须输出“为什么它是竞品”“它强在哪里”“它会在哪个决策阶段抢走用户”“我们该怎么应对”。
2. 应对建议必须标记为分析建议，不得新增无证据事实。
3. 四段式解释必须绑定 Claim 和 Evidence 引用。

验证测试：

1. 后端单元测试确认每条关键边都有四段解释。
2. 后端单元测试确认每段解释都有可追溯 Claim 或 Evidence。
3. QA 测试确认解释中无证据事实会被风险标记。

### 步骤 12：升级产品画像后端数据为横向对比

执行指令：

1. 在现有目标产品画像基础上增加目标产品、最高威胁直接竞品、最高威胁替代竞品的对比数据。
2. 第一屏对比维度包含价格带、核心卖点、主要人群、使用场景和证据可信状态。
3. 每个维度输出目标产品状态：优势、持平、短板或证据不足。
4. 功能树、评论痛点和截图证据继续作为下钻数据，不放在默认第一层。

验证测试：

## 5. 最终验收清单

交付 2.0 前必须全部满足：

1. 创建任务后默认进入竞争态势总览。
2. 总览页首屏展示一句话判断、决策可用状态、关键竞品缩略图、首要行动建议和证据风险提示。
3. 总览页支持价格带、人群、使用场景切片联动。
4. 每个关键结论都能下钻到证据和过程。
5. 竞争图谱默认展示 3 到 5 条关键关系，并提供展开全部关系能力。
6. 每条关键竞争关系展示威胁等级、PM 关系标签、入选理由、四段式解释、应对建议和证据可信状态。
7. 产品与竞品画像页支持目标产品和核心竞品横向对比。
8. 人工修正仍是受控结构化 Human Feedback，不允许自由编辑整份报告。
9. 报告页提供工作台视图和打印视图。
10. 前端不再展示 Markdown 导出入口。
11. 后端提供真实 `.docx` 导出。
12. Word 报告包含封面、目录、正文、图片、简化关系图和附录。
13. 图片缺失时写“暂无可靠图片”，不导致导出失败。
14. 证据与过程追踪页默认按结论组织证据链。
15. QA 打回、证据补齐、Analysis 重算和人工修正可追溯。
16. 技术详情默认折叠，展开后仍做脱敏。
17. 全站用户可见文案中文化，主界面不裸露英文技术词。
18. 导出报告、Trace、日志、截图和错误响应不包含 API Key 或未脱敏隐私。
19. 不引入未批准的新技术栈或复杂基础设施。
20. 后端全测、前端全测、E2E、Lint、格式检查、构建验证全部通过。

## 6. 禁止事项

1. 禁止为了 2.0 视觉改版削弱 LangGraph 真实 DAG。
2. 禁止把 QA 打回改成前端假展示。
3. 禁止在 Writer 或报告导出中新增无证据事实。
4. 禁止让前端从多个旧接口临时拼出总览主数据。
5. 禁止继续把 Markdown 作为正式交付入口。
6. 禁止让 Word 导出失败影响网页报告阅读。
7. 禁止把技术字段、ID、Agent 名称和 Evidence ID 放在主界面默认阅读层。
8. 禁止把测试 fixture 当作最终 Demo 数据。
9. 禁止改动冻结 Demo 基线而不更新冻结文档和回归测试。
10. 禁止在代码、文档、日志、Trace、截图或导出报告中写入真实 API Key。
