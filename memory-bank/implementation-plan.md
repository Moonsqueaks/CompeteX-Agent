# 竞品分析与竞争关系重建多 Agent 协作系统实施计划

## 1. 使用说明

本文档位于 `memory-bank/`，基于同目录下的 `design-document.md` 和 `tech-stack.md` 编写，用于指导 AI 开发者按小步迭代完成 MVP。文中 `backend/`、`frontend/`、`data/`、`docs/`、`demo/` 等路径均相对于项目根目录。

执行规则：

1. 每一步都必须小而具体，不跨多个责任边界。
2. 每一步都必须先明确测试，再完成实现。
3. 每一步完成后必须运行对应验证测试，并记录是否通过。
4. 严禁跳过 QA 打回、证据链、Trace、Markdown 导出这些 MVP 核心能力。
5. 严禁引入未在 `tech-stack.md` 推荐的复杂基础设施，例如 Celery、Redis、PostgreSQL、Next.js、Redux、Tailwind。
6. 本计划只写实施指令，不包含任何代码片段。

推荐执行顺序：

1. 基础工程与契约
2. 后端 Schema、存储、API
3. Agent DAG 与 QA 闭环
4. 前端页面与数据展示
5. 前后端集成
6. 演示稳定性与验收

已确认执行决策：

1. Demo 数据由用户提供真实脱敏 SKU 快照；开发者可以为测试创建独立 fixture，但不得把模拟数据当作最终演示数据。
2. MVP 目标产品固定；如用户未额外指定，开发者从用户提供的脱敏 SKU 中选择证据最完整、最适合作为演示主线的产品作为默认 Demo 产品。
3. Doubao 模型调用是可选增强；未配置模型 API Key 时，系统必须仍能依靠本地快照和规则流程跑通完整 Demo。
4. QA 打回案例优先演示“补齐一条缺失证据”，并在 Trace 中保留补齐前后的差异。
5. SQLite 存储采用“任务表 + Artifact JSON + 日志表”的轻量方案，不拆复杂关系表。
6. 产品运行时使用后台任务和前端轮询；测试中必须提供同步执行入口，保证工作流可稳定验证。
7. 任务状态枚举固定为 `created`、`collecting`、`analyzing`、`reviewing`、`writing`、`completed`、`failed`、`partial_failed`、`human_reviewing`。
8. 创建任务后前端默认跳转到过程追踪页。
9. Human Review 提交后立即触发 Analysis 局部重算；重算失败时保留反馈，并通过任务元数据标记需要重新分析。
10. `snapshot_plus_live` 在 MVP 中作为增强模式占位：前端展示说明，后端记录该模式并在 Trace 中说明已降级使用本地快照，不做真实外部采集。
11. Markdown 报告接口返回 Markdown 内容，同时保存到 `<project-root>/data/reports/`。
12. `architecture.md` 和 `progress.md` 暂不维护，等待后续指令。

## 1.1 协作提交约定

为保证多人协作期间 Demo 可持续运行、文档与代码不漂移，补充以下协作约定：

1. `main` 只保留可运行、可演示、通过当前相关测试的版本，不直接提交未完成实验代码。
2. 所有开发改动默认在独立分支完成，例如 `feat/snapshot-loader`、`fix/qa-trace-diff`、`docs/collaboration-rules`，再通过 Pull Request 合并回 `main`。
3. 分支合并前，提交人至少完成本次改动对应的测试、自查影响范围，并在 PR 描述中写明改动内容、验证结果和风险点。
4. `memory-bank/` 视为与代码同等重要的交付物；凡是改动了接口契约、流程边界、Demo 数据约定、默认目标产品、QA 打回逻辑或演示路径，必须同步更新对应文档。
5. `data/snapshots/demo_sku_snapshot.json`、`sku_01` 的 QA 打回样例、`sku_02` 的默认演示目标视为当前 Demo 基线，不得随意改动。
6. 如确需调整上述 Demo 基线，必须同时记录变更原因、影响范围、相关测试更新和新的演示口径，避免队友在不知情的情况下继续使用旧脚本或旧答辩话术。

## 2. 角色分工建议

| 角色 | 负责范围 |
|---|---|
| 后端 AI 开发者 | FastAPI、Pydantic Schema、SQLite、LangGraph、Agent、QA、Markdown 导出 |
| 前端 AI 开发者 | React、Ant Design、React Flow、页面、轮询、Human Review 交互 |
| 集成 AI 开发者 | API 契约、OpenAPI 类型同步、端到端流程、Playwright 验证 |
| 文档与演示 AI 开发者 | Demo 数据检查、验收清单、演示脚本、风险兜底 |

## 3. 实施步骤

### 步骤 01：建立项目骨架

执行指令：

1. 按 `tech-stack.md` 的目录建议创建 `backend`、`frontend`、`data`、`docs`、`demo` 目录，并保留项目根目录下的 `memory-bank` 作为项目记忆文档目录。
2. 后端只创建基础应用入口、测试目录和配置文件。
3. 前端只创建 Vite React TypeScript 基础项目。
4. 不实现任何业务逻辑。

验证测试：

1. 后端基础测试套件可以启动并通过空项目健康检查。
2. 前端基础测试套件可以启动并通过默认渲染检查。
3. 前端构建可以成功完成。
4. 目录结构与 `tech-stack.md` 推荐目录一致。

### 步骤 02：配置代码质量工具

执行指令：

1. 配置后端 Ruff、Pytest、测试环境变量加载方式。
2. 配置前端 ESLint、Prettier、Vitest、Testing Library。
3. 保证格式化和 lint 不依赖真实 API Key。

验证测试：

1. 后端 lint 检查通过。
2. 后端测试套件通过。
3. 前端 lint 检查通过。
4. 前端测试套件通过。
5. 前端构建通过。

### 步骤 03：定义后端统一响应与错误格式

执行指令：

1. 建立统一 API 响应结构，包含 `data`、`error`、`trace_id`。
2. 建立统一错误对象，包含错误码、错误消息和错误详情。
3. 所有后续 API 都必须使用该响应结构。

验证测试：

1. 新增 API 响应格式测试，确认成功响应包含 `data` 和 `trace_id`。
2. 新增错误响应测试，确认失败响应包含错误码和错误消息。
3. 确认响应中不会出现 API Key、模型密钥或环境变量原文。

### 步骤 04：定义核心 Pydantic Schema

执行指令：

1. 实现 `AnalysisTask`、`AgentMessage`、`Product`、`FeatureTree`、`PricingModel`、`UserPersona`。
2. 实现 `Evidence`、`Claim`、`CompetitionEdge`、`ReviewTask`、`HumanFeedback`。
3. 实现 `AgentRunLog`、`ToolCallLog`、`TokenUsageLog`。
4. 字段命名统一使用 `snake_case`。

验证测试：

1. 为每个核心 Schema 增加合法样例校验测试。
2. 为必填字段缺失增加失败测试。
3. 为 `Claim` 缺少证据 ID 增加风险校验测试。
4. 为 `CompetitionEdge` 分数范围增加边界测试。
5. 确认 Schema 可以生成 OpenAPI 文档。

### 步骤 05：建立 SQLite 存储层

执行指令：

1. 建立 SQLite 数据库连接和初始化流程。
2. 建立任务表、Artifact JSON 表、Trace/运行日志表、人工反馈表。
3. 建立 Repository 层，禁止 API 路由直接操作数据库。

验证测试：

1. 使用临时 SQLite 数据库运行存储层测试。
2. 验证任务可以创建、读取、更新状态。
3. 验证 Evidence、Claim、CompetitionEdge 可以作为 Artifact JSON 保存和查询。
4. 验证 Trace 日志可以按任务 ID 查询。
5. 验证测试结束后不会污染真实数据目录。

### 步骤 06：准备本地 Demo 快照数据规范

执行指令：

1. 在 `<project-root>/data/snapshots` 定义自动猫砂盆 SKU 快照文件结构。
2. 等待用户提供真实脱敏 SKU 快照作为最终 Demo 数据；如需先开发测试，只能在测试目录创建独立 fixture。
3. 每个 SKU 至少包含名称、品牌、价格、卖点、评论摘要、来源说明。
4. 固定保留一个缺少价格访问时间或截图路径的 SKU，用于 QA 打回演示。

验证测试：

1. 增加快照格式校验测试，确认最终 Demo 快照至少存在 8 个 SKU。
2. 验证每个 SKU 都能转换为 `Product` 和 `Evidence`。
3. 验证用于 QA 打回的 SKU 确实缺少指定证据字段。
4. 验证快照中不包含手机号、真实账号 ID、API Key 或敏感隐私。

### 步骤 07：实现快照加载服务

执行指令：

1. 实现 Snapshot Loader，负责读取本地 SKU 快照和评论快照。
2. 将快照转换为标准 `Product`、`Evidence`、`ReviewInsight`。
3. 当快照缺字段时保留缺失状态，不要自动编造数据。

验证测试：

1. 验证 Snapshot Loader 可以读取所有 Demo SKU。
2. 验证缺失字段会被保留为缺失或“暂无可靠数据”。
3. 验证非法快照文件会返回可诊断错误。
4. 验证加载结果可以通过 Pydantic Schema 校验。

### 步骤 08：实现任务创建 API

执行指令：

1. 实现 `POST /tasks`。
2. 支持目标产品名称、商品链接、品类、子类、数据模式、用户研究文本。
3. 创建任务后写入数据库，并返回任务 ID。
4. 暂时只创建任务，不启动完整 Agent 流程。
5. 如请求未指定目标产品，默认选择最终 Demo 快照中证据最完整的目标 SKU。

验证测试：

1. API 测试确认合法请求可以创建任务。
2. API 测试确认缺少目标产品名称时返回校验错误。
3. API 测试确认默认数据模式为本地快照模式。
4. API 测试确认返回结构符合统一响应格式。
5. API 测试确认未指定目标产品时会选择默认 Demo 目标产品。

### 步骤 09：实现任务状态查询 API

执行指令：

1. 实现 `GET /tasks/{task_id}`。
2. 返回任务基础信息、当前状态、创建时间、更新时间。
3. 不返回完整报告或大型 Trace 数据。

验证测试：

1. API 测试确认已存在任务可以查询。
2. API 测试确认不存在任务返回标准错误。
3. API 测试确认任务状态字段只使用允许枚举值。
4. API 测试确认响应不包含敏感配置。

### 步骤 10：建立 LangGraph 状态对象

执行指令：

1. 定义 `TaskGraphState` 的状态字段。
2. 状态必须覆盖任务、产品、证据、结论、竞争边、QA 记录、人工反馈、Agent 消息和运行日志。
3. 保持状态对象轻量，复杂校验交给 Pydantic Artifact。

验证测试：

1. 单元测试确认初始状态可以从任务生成。
2. 单元测试确认状态可以追加 Product、Evidence、Claim、ReviewTask。
3. 单元测试确认状态序列化后可以用于 Trace 展示。
4. 单元测试确认缺少任务 ID 时状态初始化失败。

### 步骤 11：实现 Collection Agent 节点

执行指令：

1. Collection Agent 只负责读取本地快照和用户研究文本。
2. 输出标准 Product、Evidence 和 ReviewInsight。
3. 不进行竞品评分，不生成最终报告。
4. 对缺失字段保留缺失状态，并写入 Trace。

验证测试：

1. Agent 单元测试确认输入任务后可以生成至少 8 个 Product。
2. Agent 单元测试确认每个 Product 至少有一个 Evidence 关联。
3. Agent 单元测试确认缺失访问时间的证据不会被自动补齐。
4. Trace 测试确认 Collection Agent 运行日志被记录。

### 步骤 12：实现评分服务

执行指令：

1. 实现 CompetitionEdgeScore 的五维规则评分。
2. 评分维度包括需求替代性、上下文匹配度、决策阶段影响力、证据置信度、市场信号强度。
3. 每个维度必须输出可解释分数。
4. 不使用复杂预测模型。

验证测试：

1. 单元测试确认总分由五个维度按权重计算。
2. 单元测试确认每个维度分数必须在合法范围内。
3. 单元测试确认低证据置信度会降低总分。
4. 单元测试确认不同切片会产生不同竞争边排序。

### 步骤 13：实现 Analysis Agent 节点

执行指令：

1. Analysis Agent 根据 Product 和 Evidence 生成 FeatureTree、PricingModel、UserPersona。
2. 实现同类相似、需求替代、内容共现三路竞品召回的 MVP 规则。
3. 生成 CompetitionEdge 和 Claim。
4. 每条核心 Claim 必须绑定证据 ID 或标记为待 QA 风险。

验证测试：

1. Agent 单元测试确认可以生成目标产品画像。
2. Agent 单元测试确认可以生成直接竞品和替代竞品。
3. Agent 单元测试确认每条 CompetitionEdge 包含切片、决策阶段和评分解释。
4. Agent 单元测试确认没有证据的 Claim 会进入风险状态。
5. Trace 测试确认 Analysis Agent 运行日志被记录。

### 步骤 14：实现 QA 规则服务

执行指令：

1. 实现证据完整性检查。
2. 实现价格、评分、评价数、排名的访问时间检查。
3. 实现关键价格或认证信息的截图路径检查。
4. 实现推断标注、敏感表达、评论聚类、前后矛盾检查。
5. 检查结果输出 ReviewTask。

验证测试：

1. 单元测试确认缺少证据 ID 的 Claim 会被标记。
2. 单元测试确认缺少价格访问时间会触发打回 Collection。
3. 单元测试确认推断内容未标注会触发打回 Analysis。
4. 单元测试确认单条评论被过度概括会触发风险。
5. 单元测试确认合格 Claim 可以通过 QA。

### 步骤 15：实现 QA Agent 节点

执行指令：

1. QA Agent 调用 QA 规则服务审查 Claim、Evidence 和 CompetitionEdge。
2. 审查通过时输出通过状态。
3. 审查失败时输出结构化 `revision_request`。
4. 打回目标只能是 Collection Agent、Analysis Agent 或 Writer Agent。

验证测试：

1. Agent 单元测试确认合格数据会进入通过状态。
2. Agent 单元测试确认缺少价格访问时间会生成打回 Collection 的消息。
3. Agent 单元测试确认分析矛盾会生成打回 Analysis 的消息。
4. Trace 测试确认 QA 检查项、问题等级和打回目标被记录。

### 步骤 16：实现 QA 打回后的 Collection 修复逻辑

执行指令：

1. Collection Agent 接收 QA 的补采请求。
2. 对 Demo 快照中可补齐的数据补充证据字段。
3. 对无法补齐的数据写入“暂无可靠数据”。
4. 修复结果必须形成新的 Evidence 或更新原 Evidence，并保留前后差异。

验证测试：

1. 集成测试确认 QA 打回 Collection 后会再次执行 Collection。
2. 集成测试确认可补齐证据会更新访问时间或截图路径。
3. 集成测试确认不可补齐证据会明确标记“暂无可靠数据”。
4. Trace 测试确认打回前后差异可查询。

### 步骤 17：实现 QA 打回后的 Analysis 重算逻辑

执行指令：

1. Analysis Agent 接收 QA 的重分析请求。
2. 针对被打回的 Claim 或 CompetitionEdge 重新计算。
3. 重算后必须更新评分解释和风险状态。
4. 不影响无关竞品边。

验证测试：

1. 集成测试确认 QA 打回 Analysis 后会再次执行 Analysis。
2. 集成测试确认被打回 Claim 的状态会变化。
3. 集成测试确认 CompetitionEdgeScore 会根据新证据重新计算。
4. 集成测试确认无关竞争边保持稳定。

### 步骤 18：组装 LangGraph 主流程

执行指令：

1. 将 Collection、Analysis、QA、Writer 节点组装成 LangGraph 工作流。
2. 配置 QA 通过到 Writer 的条件边。
3. 配置 QA 打回 Collection 和 Analysis 的条件边。
4. 设置最大打回次数，避免无限循环。

验证测试：

1. 工作流测试确认正常数据可以从创建任务走到完成。
2. 工作流测试确认缺失证据会触发 QA 打回。
3. 工作流测试确认打回后可以继续走到完成或可诊断失败。
4. 工作流测试确认超过最大打回次数会进入失败状态并保留 Trace。

### 步骤 19：实现 Writer Agent 节点

执行指令：

1. Writer Agent 只接收 QA 通过或带风险标记的结构化产物。
2. 生成网页报告数据结构。
3. 生成报告摘要、产品画像、竞品发现、动态切片、用户研究、建议和 Evidence 索引。
4. 不在 Writer Agent 中编造新的事实字段。

验证测试：

1. Agent 单元测试确认报告包含所有必需章节。
2. Agent 单元测试确认报告中的核心结论都能追溯 Evidence。
3. Agent 单元测试确认风险 Claim 会在报告中标明。
4. Trace 测试确认 Writer Agent 运行日志被记录。

### 步骤 20：实现 Markdown 报告导出服务

执行指令：

1. 使用模板生成 Markdown 报告。
2. Markdown 必须包含 Claim、Evidence、置信度、访问时间和推断标识。
3. Markdown 不导出 API Key、完整系统 Prompt、未脱敏隐私文本。
4. 导出文件保存到本地报告目录，并记录元信息。

验证测试：

1. 单元测试确认 Markdown 包含九个报告章节。
2. 单元测试确认每条核心 Claim 后都能看到 Evidence 索引。
3. 单元测试确认缺失证据的数据会显示“暂无可靠数据”。
4. 安全测试确认 Markdown 不包含敏感环境变量或 API Key 模式。

### 步骤 21：实现报告 API

执行指令：

1. 实现 `GET /tasks/{task_id}/report`。
2. 实现 `GET /tasks/{task_id}/report/markdown`。
3. 未完成任务请求报告时返回标准错误。
4. 导出失败时保留网页报告可用。

验证测试：

1. API 测试确认完成任务可以获取网页报告数据。

## 5. 禁止事项

1. 禁止为了赶进度绕过 Pydantic Schema。
2. 禁止 Agent 之间只传自然语言，不传结构化 Artifact。
3. 禁止在 Claim 中写没有 Evidence 的强事实结论。
4. 禁止 QA 打回只做前端假展示。
5. 禁止把真实 API Key 写入代码、文档、日志、Trace 或截图。
6. 禁止引入超出 MVP 的复杂基础设施。
7. 禁止在 Writer Agent 中编造价格、认证、尺寸、销量、排名。
8. 禁止让前端直接依赖临时字段名，接口变更必须同步类型和测试。
9. 禁止在第 15 天后新增大功能，只允许修稳定性、体验和表达。
